# 产品需求文档：基于 MCP 与 LangGraph 的智能出行助手 Agent 系统

## 1. 产品概述

### 1.1 产品背景
现代出行场景中，用户需要整合天气、路况、导航、行程记录等多维信息。传统出行应用功能分散，用户需频繁切换 App 手动查询与规划。大型语言模型（LLM）的 Agent 能力结合标准化工具协议，可构建一站式智能出行助手，通过自然语言对话自动调用外部服务，完成“查询-规划-记录”闭环。

### 1.2 产品定位
基于 **Model Context Protocol (MCP)** 与 **LangChain/LangGraph** 框架，连接阿里通义千问等 LLM，提供可扩展的工具集合（天气、地图、文件系统等）的智能出行助手 Agent 系统。用户可通过自然语言交互完成出行相关的信息检索、路线规划、行程记录保存等任务。

### 1.3 产品目标
- **核心目标**：实现 LLM 通过 MCP 标准协议无缝调用外部工具，完成多步骤出行任务。
- **体验目标**：对话式交互，用户无需学习命令或界面操作；响应时间 < 5 秒（含工具调用）。
- **技术目标**：基于 LangGraph 构建可控、可观测的 Agent 工作流，支持工具热插拔与状态管理。

## 2. 用户角色

| 角色 | 描述 | 典型场景 |
|------|------|----------|
| 普通出行用户 | 需要查询天气、导航、记录行程的日常用户 | 周末出行前：“帮我查一下杭州周六的天气，然后规划从西湖到灵隐寺的驾车路线，并把行程保存到本地” |

## 3. 核心功能需求

### 3.1 自然语言对话与意图识别
- 用户通过文本输入需求，Agent 理解出行相关意图（查询、规划、记录、组合任务）。
- 支持多轮对话，维护上下文（如出发地、目的地、偏好日期）。
- 意图不明确时主动引导（例如：“请问您要查询哪座城市的天气？”）。

### 3.2 MCP 工具集成
系统通过 MCP 标准协议连接以下工具（每个工具均实现 MCP Server 接口，Agent 作为 MCP Client 调用）：

#### 3.2.1 天气查询工具
- **功能**：根据城市/地点和日期（今天、未来几天）返回天气状况（温度、湿度、降水概率、风力、预警）。
- **数据源**：可接入第三方天气 API（如和风天气、OpenWeatherMap）。
- **MCP 接口示例**：`get_weather(city, date)` → JSON 格式天气数据。

#### 3.2.2 地图导航工具
- **功能**：查询两地之间的路线（驾车、公交、步行、骑行），返回预估时间、距离、关键转向点、实时路况（如可用）。
- **数据源**：高德地图 / 百度地图 / Google Maps API。
- **支持参数**：出发地、目的地、交通方式、出发时间或到达时间。
- **MCP 接口示例**：`get_directions(origin, destination, mode, departure_time)` → 路线摘要 + 步骤列表。

#### 3.2.3 文件写入工具
- **功能**：将出行规划、路线描述、天气信息等文本内容写入本地文件（如 Markdown、TXT）。
- **支持操作**：创建新文件、追加内容、指定保存路径。
- **安全限制**：仅允许写入特定工作目录（如 `./travel_plans/`），防止路径遍历攻击。
- **MCP 接口示例**：`write_file(filename, content, append=False)` → 成功/失败状态。

#### 3.2.4 （可选扩展）日历提醒工具
- **功能**：将出行事件添加到系统日历（如 iCal、Google Calendar），设置提醒时间。

### 3.3 多步规划与 Agent 编排 (LangGraph)
- 系统使用 LangGraph 构建状态图，支持：
  - **顺序执行**：例如先查天气，再根据天气推荐是否驾车，然后规划路线。
  - **条件分支**：如下雨则推荐公共交通并提示带伞。
  - **循环/重试**：工具调用失败时自动重试或询问用户。
  - **人机协同**：关键步骤（如写入文件前）可请求用户确认路径。
- Agent 内存：维护对话状态与已获取的工具结果，避免重复调用。

### 3.4 结果展示与输出
- 对话界面中，以自然语言总结工具返回结果（例如：“杭州周六晴天，22~28°C，适合出行。从西湖到灵隐寺驾车约25分钟，路线已保存到 `travel_plan.md`。”）
- 支持结构化展示（如天气卡片、路线缩略图，取决于前端能力，PRD 暂不限）。
- 文件写入后，返回文件路径及可点击链接（如果是本地环境）。

## 4. 非功能需求

### 4.1 性能要求
- Agent 单次决策 + 工具调用链（含一次 LLM 调用）总耗时 < 5 秒（不含外部 API 网络延迟）。
- 支持并发用户数：初期 50 QPS，可水平扩展。
- 工具调用超时设置：每个外部 API 请求超时 3 秒，超时后 Agent 应友好提示。

### 4.2 可靠性
- 工具调用失败重试机制：最多重试 2 次（如网络抖动）。
- Agent 工作流异常中断时，恢复并告知用户错误原因，不丢失已获取信息。
- 文件写入确保原子性：写入失败时数据不损坏。

### 4.3 安全性
- LLM API 密钥、地图/天气 API 密钥通过环境变量或密钥管理服务注入，不写入代码或日志。
- 文件写入工具限定目录，禁止访问系统敏感路径（如 `/etc`、`~/.ssh`）。
- 用户输入需经过注入检查（防止试图通过 Prompt 注入操纵文件写入路径）。

### 4.4 可扩展性
- 新增工具只需实现 MCP 协议并注册到 MCP Server，Agent 通过动态发现即可调用，无需修改核心代码。
- 支持更换底层 LLM（如从通义千问切换到 GPT-4），通过 LangChain 的抽象层实现。

### 4.5 可观测性
- 记录每次 Agent 执行的完整轨迹（用户输入 → 意图 → 调用的工具序列 → 参数 → 结果 → 最终回复）。
- 关键指标：工具调用延迟、成功率、LLM token 消耗。

## 5. 技术架构概述

### 5.1 整体架构图（文字描述）
```
[用户] ↔ [前端界面 (CLI/Web)] ↔ [Agent 服务 (FastAPI/WebSocket)]
                                           ↓
                                    LangGraph Agent
                                    (状态机 + LLM 决策)
                                           ↓
                                    MCP Client (内置)
                                           ↓
                              MCP Server (工具注册中心)
                               ↓       ↓           ↓
                         天气工具   地图工具   文件工具
                          (调用    (调用       (本地
                           外部API)  外部API)     文件系统)
```

### 5.2 核心组件说明
- **LLM**：阿里通义千问 (Qwen-Max) 或其他，通过 LangChain ChatModel 统一接口调用。
- **LangGraph**：构建 Agent 工作流。定义状态（State）包含 `messages`（对话历史）、`tool_results`（缓存工具输出）、`next_step`（决策节点）。节点包括：`agent_think`（LLM 决策调用哪个工具）、`tool_executor`（并行/串行执行 MCP 工具）、`final_response`（生成最终回复）。
- **MCP 实现**：使用开源 MCP Python SDK（如 `mcp` 库）。每个工具封装为 MCP Tool，定义输入 Schema（JSON Schema）、输出格式。Agent 通过 `mcp.Client` 动态列出工具并调用。
- **工具适配层**：若第三方 API 为非标准协议，编写轻量包装器使其符合 MCP 规范。

### 5.3 数据流示例（查天气 + 规划路线）
1. 用户输入：“杭州周六天气怎么样？好的话帮我规划从浙大到西湖的公交路线。”
2. Agent 调用 LLM（带工具描述）→ LLM 决定先调用 `get_weather`。
3. Agent 通过 MCP Client 调用天气工具 → 返回天气数据。
4. Agent 将天气结果放入状态，再次调用 LLM → LLM 判断天气良好，决定调用 `get_directions`（mode=transit）。
5. 调用地图工具 → 返回路线。
6. Agent 生成最终回复：“杭州周六晴，适合公交出行。浙大到西湖公交约30分钟，需换乘1次...”

## 6. 用户交互流程示例

### 6.1 典型对话流程（文本界面）
```
用户: 我下周一下午2点要去北京首都机场，从国贸出发，查一下那天天气，顺便规划驾车路线，并把行程保存到文件。

Agent: 好的，正在为您查询下周一（2025-01-15）北京国贸至首都机场的天气和驾车路线。
       [调用天气工具] → 返回：多云，-4°C~3°C，北风3级，无雨雪。
       [调用导航工具] → 路线：约60分钟，经机场高速，预计25公里。
       已为您生成行程摘要并保存到文件：/travel_plans/beijing_trip_20250115.md。
       是否需要添加日历提醒？

用户: 好的，加一个出发前两小时的提醒。

Agent: [调用日历工具] → 已添加提醒：2025-01-15 12:00 提醒“出发前往首都机场”。
```

### 6.2 异常处理
```
用户: 帮我查一下火星的天气。

Agent: 抱歉，天气查询工具仅支持地球上的城市。请提供有效城市名称，例如“北京”或“纽约”。
```

## 7. 接口定义（MCP 工具规范）

每个工具提供以下 MCP 描述（示例用 JSON Schema 表示）：

### 7.1 天气工具
```json
{
  "name": "get_weather",
  "description": "获取指定城市在指定日期的天气信息",
  "inputSchema": {
    "type": "object",
    "properties": {
      "city": {"type": "string", "description": "城市名称，如北京"},
      "date": {"type": "string", "format": "date", "description": "日期，YYYY-MM-DD，默认今天"}
    },
    "required": ["city"]
  }
}
```
输出示例：
```json
{
  "city": "杭州",
  "date": "2025-01-20",
  "condition": "晴",
  "temp_high": 28,
  "temp_low": 22,
  "humidity": 65,
  "precip_prob": 10,
  "wind": "东南风2级"
}
```

### 7.2 地图导航工具
```json
{
  "name": "get_directions",
  "description": "规划两地之间的路线",
  "inputSchema": {
    "type": "object",
    "properties": {
      "origin": {"type": "string"},
      "destination": {"type": "string"},
      "mode": {"type": "string", "enum": ["driving", "transit", "walking", "bicycling"]},
      "departure_time": {"type": "string", "format": "date-time", "description": "可选，出发时间"}
    },
    "required": ["origin", "destination"]
  }
}
```

### 7.3 文件写入工具
```json
{
  "name": "write_file",
  "description": "将内容写入指定文件（仅限 ./travel_plans/ 目录）",
  "inputSchema": {
    "type": "object",
    "properties": {
      "filename": {"type": "string", "description": "文件名，不支持路径分隔符"},
      "content": {"type": "string"},
      "append": {"type": "boolean", "default": false}
    },
    "required": ["filename", "content"]
  }
}
```

## 9. 成功指标（KPI）

- 任务完成率：用户发起的多步出行任务（含至少一次工具调用）成功率达到 95% 以上。
- 平均响应时间：从用户输入到最终回复 < 5 秒（不含 LLM 首 token 生成时间）。
- 工具调用覆盖率：至少支持天气、地图、文件三个核心工具，每个工具可用性 99.9%。
- 用户满意度：通过对话后隐式反馈（例如无帮助纠正）预估，目标 < 5% 的否定交互。

## 核心发现

### 1. 可行性是最大收益点（+8%）

V2 的 `tech_advisor` 专门负责技术栈推荐、数据模型设计、API 端点和风险评估，而 V1 的 `architecture_design` 阶段更偏向概要设计。这在"校园二手交易"样本上体现最明显：V1 可行性仅 6 分，V2 提升到 9 分（+50%）。

**Why**: 多智能体将技术评估从"一个阶段"升级为"一个独立专家 Agent"，提示词更聚焦、输出更深入。

### 2. 反思循环有效但存在天花板

所有 3 个样本初始分均低于 80 阈值（65/78/78），经 1 轮修订后全部达标（80/82/82），平均提升 +7.6 分。但最终分全部集中在 80-82 区间，说明：

- 反思机制能修复明显的质量问题（如 JSON 解析失败、章节缺失）
- 但无法突破模型本身的能力上限（deepseek-v4-pro 的推理深度）
- 修订轮次 1 轮即可达标，说明 max_rounds=2 的设置合理

### 3. 相关性略降（-3%）

V2 生成的内容更丰富（PRD 长度比 V1 长 36-46%），但有时会引入偏离核心产品想法的内容。例如"职业规划"样本从 10 降到 9——tech_advisor 可能过度展开通用技术方案而偏离具体场景。

**启示**: 可以在 Agent prompt 中加强"紧扣产品想法"的约束，或让 Reviewer 将相关性也纳入重点检查维度。

### 4. 代价：2.7 倍耗时

| 版本 | LLM 调用次数 (估算) | 实际耗时 |
|------|---------------------|----------|
| V1 | 4 次 (并行分析×2 + 流程设计 + 文档合成) | 333s |
| V2 | 10-14 次 (4 Agent×2轮 + Reviewer×2 + 修订路由 + 文档合成) | 902s |

耗时增加主要来自反思循环中的二次生成。对于时间敏感的场景，可以考虑提升初始 prompt 质量以减少触发反思的概率。

### 5. tech_advisor JSON 解析是已知技术债务

3 个样本中 tech_advisor 均出现 JSON 解析失败：
- 样本 1: 首轮解析失败，修正也失败（降级为 `{}`），修订轮成功
- 样本 2: 首轮解析失败，修正成功
- 样本 3: feature_planner 修订轮解析失败，修正成功

根因是 tech_advisor 输出最长（10k-15k 字符），复杂 JSON 结构容易出现格式错误。当前三级防御（JSON Mode → 7规则修复 → LLM 自修正）覆盖了大部分情况，但 tech_advisor 的 prompt 可以进一步优化以降低初始失败率。

### 6. 反思路由器智能过滤有效

样本 2 中，revision_router 仅选择了 3 个 Agent 修订（跳过了 requirements_analyst），避免了不必要的重复生成。这说明 Reviewer 的逐 Agent 反馈机制是有效的。

---

## 结论

多智能体架构在**技术可行性评估**上有明确优势（+8%），反思循环能可靠地将低质量 PRD 修复到 80 分以上。但整体质量提升幅度有限（综合分 +0.1），当前质量天花板主要取决于底层 LLM 模型能力而非编排模式。

**建议**:
- 对技术方案要求高的场景（B2B、IoT、金融等），使用 V2 多智能体模式
- 对快速原型阶段，V1 经典模式性价比更高（5.5min vs 15min）
- 升级底层模型（如 deepseek-v4-pro → claude-sonnet-4-6）预期可同时提升两个版本的绝对分数

## 一、项目背景与目标

### 1.1 背景与痛点
- **痛点1**：产品经理撰写PRD时，经常面临“从0到1”的结构混乱，不同项目文档格式差异大，缺乏统一标准。
- **痛点2**：直接使用通用大模型（如ChatGPT）生成的PRD存在“幻觉”，缺乏行业真实案例支撑，且无法复用企业内部优秀文档沉淀。
- **痛点3**：撰写PRD是一个多阶段思考过程（需求→架构→流程→细节），现有AI工具只提供单轮对话，无法模拟产品经理的真实工作流。

### 1.2 项目目标
- **核心目标**：打造一个基于 **RAG + 链式工作流** 的智能PRD生成平台，输入一句话产品想法，输出符合大厂规范的完整PRD文档。
- **关键结果（KRs）**：
  - KR1：PRD生成时间 ≤ 3分钟（从输入到输出）。
  - KR2：生成的PRD在结构完整度（≥90%字段覆盖）和格式正确率（100%符合Markdown/JSON约束）上通过人工抽检。
  - KR3：通过私有知识库检索，使核心业务描述与历史优秀案例的语义相似度 ≥ 0.75。

## 三、功能需求（MVP）

### EPIC 1：知识库管理（RAG核心）
| 功能模块 | 描述 | 优先级 |
| :--- | :--- | :--- |
| **1.1 本地知识库构建** | 上传历史PRD（Markdown/PDF/DOCX），自动切分、向量化、存入ChromaDB | P0 |
| **1.2 语义检索** | 根据用户输入的产品想法，检索最相似的3个历史PRD片段，作为生成时的参考上下文 | P0 |
| **1.3 知识库增量更新** | 支持用户将新生成的PRD一键加入知识库，实现“使用即学习” | P1 |
| **1.4 混合检索（BM25+向量）** | 提升召回准确率，避免纯语义检索漏掉关键词 | P2 |

### EPIC 2：链式工作流（LangGraph Agent）
| 功能模块 | 描述 | 优先级 |
| :--- | :--- | :--- |
| **2.1 多阶段流水线** | 依次执行：需求分析 → 架构设计 → 流程梳理 → 文档定稿 | P0 |
| **2.2 状态管理与回溯** | 每个阶段结果可查看，允许用户“重新生成上一阶段”或“微调后继续” | P1 |
| **2.3 动态Prompt注入** | 每个阶段根据前一阶段输出和知识库检索结果，自动生成专属系统提示词 | P0 |
| **2.4 流程图自动生成** | 在“流程梳理”阶段输出Mermaid格式的业务流程图/状态图 | P0 |

### EPIC 3：PRD内容规范与输出
| 功能模块 | 描述 | 优先级 |
| :--- | :--- | :--- |
| **3.1 大厂标准PRD模板** | 固定包含：版本记录、背景目标、用户画像、功能列表、非功能需求、埋点&迭代规划 | P0 |
| **3.2 结构化输出约束** | 每个阶段强制返回JSON，再用后处理转为Markdown，保证格式统一 | P0 |
| **3.3 异常&边界自动补充** | 根据功能列表，AI自动生成异常处理逻辑（如网络失败、权限不足） | P1 |
| **3.4 导出与分享** | 支持导出为Markdown文件、PDF，及生成可分享的在线链接（部署后） | P1 |

### EPIC 4：用户体验与界面（可选，若做Web）
| 功能模块 | 描述 | 优先级 |
| :--- | :--- | :--- |

## 四、技术架构（非功能性需求）

### 4.1 架构图（简述）
```
[用户输入] → [前端 Web/CLI] → [LangGraph 工作流引擎]
                                    ↓
                             [DeepSeek V4 Pro (LLM)]
                                    ↑
                        ┌───────────┴───────────┐
                   [RAG 检索器]            [Prompt 管理器]
                        ↑
                  [ChromaDB 向量库]
                        ↑
                  [本地 PRD 样本库]
```

### 4.2 技术选型
| 层 | 技术 | 理由 |
| :--- | :--- | :--- |
| 后端工作流 | LangGraph + LangChain | 原生支持多节点状态机，便于链式调用与回溯 |
| LLM | DeepSeek V4 Pro | 成本低，上下文128K，支持长PRD生成 |
| 向量数据库 | ChromaDB（本地） | 免运维，适合个人项目展示，隐私安全 |
| 嵌入模型 | `BAAI/bge-base-zh-v1.5` | 中文效果好，轻量级 |
| 前端（可选） | Streamlit / Next.js | 快速搭建演示界面 |
| 部署 | HuggingFace Spaces / Vercel | 免费，可获得公开链接 |

### 4.3 性能指标
- **PRD生成延迟**：≤ 90秒（四阶段顺序调用，含LLM推理）
- **检索延迟**：≤ 1秒（ChromaDB本地，＜1000条文档）
- **并发支持**：单用户设计，MVP不要求高并发

## 六、风险与依赖

| 风险 | 影响 | 缓解措施 |
| :--- | :--- | :--- |
| **LLM幻觉** | 生成内容包含虚假或不合理的功能 | RAG注入真实案例 + 每个阶段输出校验 + 人工可编辑覆盖 |
| **格式失控** | LLM返回不规范的JSON或乱码 | 强制function calling / JSON mode + 后验修复正则 + 降低temperature重试 |
| **知识库不足** | 样本少于10个时检索效果差 | MVP提供3个通用PRD示例，并支持用户快速添加 |
| **链式错误传播** | 第一阶段输出错误导致后续全错 | 允许用户任意阶段重试 / 手动修正输入 |
| **API成本** | 长PRD生成消耗tokens较多 | 选用DeepSeek V4 Pro（低定价） + 缓存常见模板 |
